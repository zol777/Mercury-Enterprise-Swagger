## JWT Authentication

### Overview

We use JSON Web Tokens (JWT) for API request authentication. JWTs provide a stateless, scalable authentication mechanism that allows both parties to verify the token's authenticity and integrity without additional server lookups.

Our API **only supports ES256** (ECDSA using the P-256 curve and SHA-256) for signing and verifying JWTs. The private key must be an ECDSA P-256 key encoded in PKCS#8 PEM format.

### Supported Algorithm

| Algorithm | Key Type | Key Format | Use Case | Security Level |
|-----------|----------|-----------|----------|----------------|
| ES256 | Asymmetric (ECDSA-SHA256) | ECDSA P-256, PKCS#8 PEM | All environments | High |

> HS256 and other algorithms are **not supported** by this API.

### JWT Structure

A JWT consists of three base64url-encoded parts separated by dots:

`header.payload.signature`

**Header:**

```json
{
  "alg": "ES256",
  "typ": "JWT"
}
```

**Payload (Claims):**

```json
{
  "sub": "af8b49c5d05d4f668d91a06875663892",
  "exp": 1772595910,
  "iat": 1772509510
}
```

- `sub`: API token associated with your integration.  
- `exp`: Token expiration time as a Unix timestamp in seconds.  
- `iat`: Token issued-at time as a Unix timestamp in seconds.

**Signature:** Generated by signing the encoded header and payload with your ECDSA P-256 private key in PKCS#8 PEM format using ES256.

### JWT Claims Reference

Our API expects the following claims:

| Claim | Type | Required | Description |
|-------|------|----------|-------------|
| `sub` | String | Yes | API token assigned to your integration |
| `exp` | Integer | Yes | Token expiration time (Unix timestamp, seconds) |
| `iat` | Integer | Yes | Token issued-at time (Unix timestamp, seconds) |

Additional custom claims are ignored for authorization purposes but must not conflict with standard JWT fields.

### Token Validity Period

- **Recommended validity:** 15–30 minutes.  
- **Maximum validity:** 1 hour (tokens with longer lifetimes will be rejected).  
- **Minimum validity:** 5 minutes.  
- **Refresh mechanism:** Generate new tokens as needed using your private key; refresh tokens are not supported.

## JWT Implementation

### Key Requirements

- Algorithm: ES256 (ECDSA using P-256 and SHA-256).  
- Private key: ECDSA P-256, encoded in PKCS#8 PEM format (`-----BEGIN PRIVATE KEY-----`).  
- Public key: ECDSA P-256, X.509 SubjectPublicKeyInfo in PEM format (`-----BEGIN PUBLIC KEY-----`).  
- Keys must be stored securely and never committed to source control.

### Key Generation (ECDSA P-256, PKCS#8 PEM)

You can generate a compatible key pair using OpenSSL:

```bash
# Generate EC private key (SEC1 format)
openssl ecparam -name prime256v1 -genkey -noout -out ec-private-key.pem

# Convert to PKCS#8 PEM (recommended format for private key)
openssl pkcs8 -topk8 -nocrypt   -in ec-private-key.pem   -out private-key-pkcs8.pem

# Extract public key (X.509 SubjectPublicKeyInfo, PEM)
openssl ec -in ec-private-key.pem -pubout -out public-key.pem
```

Use `private-key-pkcs8.pem` to sign JWTs and register the corresponding `public-key.pem` with us when required.

### Token Lifecycle Considerations

- Issue tokens on demand when your backend needs to call our API.  
- Avoid issuing long-lived tokens; rotate tokens frequently.  
- Monitor and alert on failed token validations and authorization errors.

## Webhook Subscriptions

### Overview

Webhooks allow your application to receive real-time notifications when specific events occur in our system. Instead of polling our API, webhooks push data to your endpoint when events happen.

When you request access to our API, you must provide a publicly accessible HTTPS webhook URL. We will deliver webhook events to this URL; there is currently no self-service registration endpoint.

### Event Types

We currently send the following webhook event types:

| Event | Description |
|-------|-------------|
| `USER_CREATED` | Indicates that a user has been created. |
| `AQT_CREATED` | Indicates that an AQT has been created. |

### Webhook Request Headers Example

The following example illustrates the typical headers you will receive with a webhook request:

```http
POST https://your-webhook-endpoint.example.com/path HTTP/1.1
Content-Type: application/json
X-Webhook-Signature: MEUCIEVPBZLEIJCOYM/bMtVZ+g1jo3crZ1l7zmD2dkIM7Ma1AiEA3w/GiWLHPA191Z113P8QplzhcZB1GC/ZpMqv15290i8=
X-Metadata: {"timestamp":"2026-02-27T10:58:55.774598335","payloadHash":"sha256=oOqJzROopmON6jIgMiowjcXIrc+cxljMq3CqfrsbDgE=","version":"V1"}
User-Agent: Apache-HttpClient/4.5.12 (Java/17.0.3)
```

> The `X-Metadata` header is a JSON string containing the timestamp, a hash of the payload, and the version of the signing scheme.

### Webhook Endpoint Requirements

Your webhook endpoint must:

1. Accept HTTP POST requests on a public HTTPS endpoint (HTTP not allowed).  
2. Process requests within **10 seconds** (timeout threshold).  
3. Return HTTP 200–299 status code on success.  
4. Verify the webhook signature before processing.  
5. Be idempotent (handle duplicate deliveries gracefully).  
6. Use a unique event identifier in the payload (for example, `eventId`) to detect and ignore duplicate events.

### Webhook Retry Policy

If your endpoint returns a **5xx** status code, we will retry delivery according to the following policy:

| Attempt | Delay |
|---------|-------|
| 1st | Immediate (original attempt) |
| 2nd | 1 minute |
| 3rd | 1 minute |
| 4th | 1 minute |
| 5th | 1 minute |

- We retry at most **4 additional times** after the initial attempt.  
- If all retries fail with 5xx responses, the event is marked as undeliverable and no further attempts are made.

## Webhook Signature Verification

### Getting the Webhook Public Key

To verify webhook signatures, you must first obtain our RSA public key.

**Endpoint:** `GET /enterprise/webhook/public_key`

**Example response:**

```json
{
  "version": "V1",
  "publicKey": "LS0tLS1CRUdJTiBQVUJMSUMgS0VZLS0tLS0KTUZrd0V3WUhLb1pJemowQ0FRWUlLb1pJemowREFRY0RRZ0FFQTh5YXpTQmZTY05ocWVVdFJIS3lESENETW9GegpINzJibkdCemZhMGVpcDZNU2VmT0lzTWdFN3FmdTk0Vll3d0JWRlZhdlJyT0RsNHBGbHh4ajlieFV3PT0KLS0tLS1FTkQgUFVCTElDIEtFWS0tLS0t"
}
```

- `version`: Signing scheme version. Currently only `"V1"` is supported.  
- `publicKey`: Base64-encoded representation of the RSA public key in PEM format.

To use this key for verification, simply Base64-decode the `publicKey` value to obtain the PEM string and keep this PEM-formatted public key available in your application.

### Security Overview

All webhooks are signed using **RSA-SHA256** digital signatures.

| Algorithm | Type | Key Type | Signature Format | Use Case |
|-----------|------|----------|------------------|----------|
| RSA-SHA256 | Asymmetric | RSA 2048-bit public/private key pair | Binary signature value, transmitted directly in `X-Webhook-Signature` | Default |

The webhook signature value is provided in the `X-Webhook-Signature` header and can be used **directly** for verification together with the PEM-formatted public key and the `X-Metadata` header.

### Signature Verification (Your Side)

For `version = "V1"`, verification is performed using:

- The PEM-formatted RSA public key obtained from `/enterprise/webhook/public_key`.  
- The `X-Metadata` header value.  
- The `X-Webhook-Signature` header value.

Your verification logic should perform the following steps:

1. **Retrieve headers**  
   - Read the `X-Metadata` and `X-Webhook-Signature` headers from the incoming request.  
   - Treat the `X-Metadata` header value as an opaque string.

2. **Prepare the public key**  
   - Ensure you have the latest PEM-formatted RSA public key obtained from the public key endpoint.  
   - Load this PEM public key using your cryptography library (typically once on startup or when the key changes).

3. **Use the metadata as signing input**  
   - Use the exact `X-Metadata` header value as the message to verify:

     ```text
     signingInput = <exact X-Metadata header value>
     ```

   - The string must match the header value byte-for-byte (no additional whitespace or formatting changes).

4. **Verify the signature**  
   - Use the PEM public key and the `SHA256withRSA` algorithm to verify the `X-Webhook-Signature` header value against `signingInput`.  
   - The `X-Webhook-Signature` header already contains the correct signature value for verification; no extra transformation step is required.  
   - If verification fails, reject the webhook.

5. **Optional timestamp validation**  
   - If you choose to parse the metadata JSON, you may extract the `timestamp` field and ensure it is within your configured tolerance window (for example, ±5 minutes) to reduce replay risk.

If any of these steps fail, you must reject the webhook request and treat it as unauthenticated.

## Implementation Examples

This section describes the expected end-to-end flow without prescribing a specific programming language.

### Typical Integration Flow

1. **Authentication:**  
   - Generate a JWT using ES256 and your ECDSA P-256 PKCS#8 private key with the required payload structure.  
   - Call our API with the JWT in the `Authorization: Bearer <token>` header.

2. **Webhook URL Provisioning:**  
   - When requesting access to our API, provide a publicly accessible HTTPS URL that will receive webhook events.  
   - Keep this URL stable and monitor it in production.

3. **Obtaining the public key:**  
   - Call `GET /enterprise/webhook/public_key` during startup or on a scheduled basis.  
   - Decode the `publicKey` field to get the PEM-formatted public key and keep it in memory or your key store.  
   - Handle `version` changes when we introduce new signing schemes.

4. **Receiving Webhooks:**  
   - Expose the webhook endpoint you provided to us and accept POST requests with JSON payloads.  
   - Log the `X-Metadata` and any event identifiers for observability.

5. **Verification:**  
   - Verify the RSA-SHA256 signature using `X-Metadata` as the signing input, the `X-Webhook-Signature` header, and the PEM public key.  
   - Optionally parse the metadata and enforce timestamp-based replay protection.  
   - Reject any requests that fail verification.

6. **Processing:**  
   - Parse the JSON payload.  
   - Apply idempotency using a unique event ID (for example, `eventId` in the payload) to avoid processing the same event twice.  
   - Perform your business logic (update records, trigger workflows, etc.).

7. **Response:**  
   - Return HTTP 2xx status when processing is accepted.  
   - For long-running work, consider enqueuing the event for asynchronous processing and returning 200 immediately.

## Security Best Practices

### 1. JWT Token Management

- Store private keys securely (e.g., environment variables, secret managers).  
- Never commit keys to version control.  
- Rotate keys periodically (recommended: every 90 days).  
- Use short token validity periods (15–30 minutes).  
- Do not log full JWTs or sensitive claims.

### 2. Webhook Security

- Always verify signatures before processing any webhook.  
- Use HTTPS only; HTTP endpoints are not supported.  
- Optionally validate timestamps and reject old webhooks.  
- Preserve the raw request body for your own auditing or business logic if needed.  
- Implement idempotency keyed by a unique event identifier.  
- Use timing-safe comparison utilities when comparing secrets or signatures.

### 3. API Communication

- Use TLS 1.2 or higher for all API requests.  
- Implement rate limiting on your side to prevent abuse.  
- Include a correlation or request ID header (e.g., `X-Request-ID`) for debugging.  
- Log security events such as failed authentications and signature verification failures.  
- Avoid sending sensitive data in query parameters; prefer request bodies.

### 4. Key Rotation Strategy

1. Generate a new key pair.  
2. Publish the new public key and support both old and new keys during a transition period.  
3. Update your clients to start using the new private key.  
4. After a defined grace period (e.g., 30 days), remove the old public key.  
5. Archive the old key for audit purposes.

## Troubleshooting

### JWT Token Verification Fails

Possible causes:

- The token has expired (`exp` is in the past).  
- The payload does not follow the required structure (`sub`, `exp`, `iat`).  
- The token was signed with the wrong key or algorithm.  
- The token has been tampered with.

Recommended checks:

- Confirm system clocks are synchronized.  
- Inspect the decoded header and payload (without trusting them) to verify fields.  
- Ensure that the correct ECDSA P-256 public key is used for verification.  
- Regenerate a token with the expected payload and test.

### Webhook Signature Verification Fails

Possible causes:

- The `X-Metadata` value used as signing input was modified or re-formatted.  
- The public key used for verification is outdated or incorrect.  
- The signature header value has been altered in transit.

Recommended checks:

- Ensure you use the **exact** `X-Metadata` header value (including whitespace) as the signing input.  
- Confirm that you correctly decoded the `publicKey` from `/enterprise/webhook/public_key` and loaded the RSA key.  
- Verify that you are using `SHA256withRSA` as the verification algorithm.  
- Refresh the cached public key and retry if you suspect key rotation.

### Webhook Endpoint Timeout

Possible causes:

- Your endpoint performs long-running operations synchronously.  
- External services or databases are slow or unavailable.  
- There is insufficient capacity (e.g., thread pool exhaustion).

Recommended approach:

- Perform signature verification quickly.  
- Enqueue the webhook payload for asynchronous processing.  
- Return a 2xx response as soon as the event is accepted into your queue.  
- Monitor processing latency and failure rates.

### "Invalid Timestamp" or Replay Concerns

Possible causes:

- Server clock is out of sync.  
- The tolerance window for accepting timestamps is too strict.  
- Requests are delayed by intermediate systems.

Recommended fixes:

- Synchronize your servers using a reliable time source.  
- Consider slightly increasing the tolerance window if your environment has higher latency.  
- If you implement timestamp checks based on metadata, ensure you parse and interpret the timestamp consistently.
